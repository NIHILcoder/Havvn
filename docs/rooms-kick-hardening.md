# Kick hardening: E2E keyring rotation + ban list (Волна 3, D7-B + M9)

> Статус: реализовано 2026-07-22. Дизайн-инварианты — сверяться перед правками
> kick/rekey/E2E-путей.

## Проблема

Кик = owner-signed ротация кода (rekey), но E2E-секрет комнаты сознательно
переживал ротацию — кикнутый навсегда сохранял возможность расшифровывать
ЛЮБЫЕ шифротексты комнаты, включая будущие. Бана не было: с новым кодом
кикнутый возвращался как ни в чём не бывало.

## Решение 1 — keyring: новый секрет после кика, старые остаются для чтения

- `room.secret` — всегда ТЕКУЩИЙ секрет: им шифруются все НОВЫЕ файлы.
- `room.prevSecrets: string[]` — прежние секреты (только расшифровка),
  новейший первым, кап 8.
- **Кик в E2E-комнате (только владелец, внутри applyLocalRekey):** текущий
  секрет уходит в prevSecrets, минтится свежий `generateRoomSecret()`, и
  новый owner-signed E2ECfg разносит ОБА (см. совместимость ниже).
- Расшифровка (`decryptOne`) перебирает `[secret, ...prevSecrets]` — GCM-тег
  сам отвергает неверный ключ.
- Ре-шифрование старых файлов НЕ делается: кикнутый их уже имел; смысл
  ротации — отрезать от БУДУЩЕГО контента.

## Совместимость: prevSecrets подписаны ОТДЕЛЬНОЙ подписью

`e2eCanonical` (v1) менять нельзя: клиент ≤2.24, получив cfg с новым полем в
канонической форме, завалил бы подпись и отверг конфиг ЦЕЛИКОМ — включая
новый текущий секрет. Поэтому:

- v1-подпись (`sig`) по-прежнему кроет `{ownerId, e2e, secret}` — старые
  клиенты принимают новый ТЕКУЩИЙ секрет и живут дальше;
- `prevSecrets` едут рядом под собственной подписью `prevSig` над
  `['th-room-e2e-prev:v1', topic, ownerId, prevSecrets]`;
- получатель, у которого prevSig не сходится/отсутствует, отбрасывает ТОЛЬКО
  keyring (лог), но принимает v1-часть — злобный участник не может ни
  подсадить фальшивые старые ключи, ни срезать их незаметно для новых
  клиентов (срез = потеря только чтения старых файлов, не текущего ключа).
- Деградация ≤2.24 после кика: новые файлы читают (новый секрет из v1-части),
  старые — только если уже скачаны (keyring им не достаётся). Честная цена.

## Решение 2 — бан-лист (без нового протокола)

- Owner-signed `rekey` уже несёт `kickedId` и уже верифицируется каждым
  участником → в `applyLocalRekey` каждый добавляет `kickedId` в
  `room.bans: Set<memberId>` (персистится).
- Ранний дроп: любое gossip-сообщение с `memberId` из bans отбрасывается ДО
  обработки — hello забаненного не воскрешает его в ростере, даже если новый
  код к нему утёк.
- Ограничение (задокументированное): бан локален у тех, кто ВИДЕЛ rekey
  (+persist). Joiner, пришедший после кика, записи не имеет — распространение
  owner-signed бан-листа через HELLO — отдельный следующий шаг. Кикнутый без
  нового кода в комнату всё равно не попадает; бан закрывает сценарий утечки
  кода.
- Разбан: не предусмотрен в этом чанке (rekey необратим по построению).

## Точки в коде

- engine: `E2ECfg.prevSecrets/prevSig`, `e2ePrevCanonical`, `signE2ECfg`,
  `verifyE2ECfg` (prev-часть опциональна), `maybeAdoptE2E` (owner-
  authoritative замена keyring'а), `applyLocalRekey` (ротация у владельца,
  бан у всех), `decryptOne` (перебор ключей), центральный ban-дроп в
  диспетчере gossip.
- manager/db: `PersistedRoom.prevSecrets/bans`, `room-e2e` и `room-rekey`
  ipc расширены, joinPayload кормит оба.
- Тест-долг: keyring-сценарии в room-e2e-adopt.test.ts (ротация при кике,
  срезанный/подделанный prevSig, decrypt-fallback) — добавить при следующем
  заходе; существующие adopt-тесты кроют регрессии v1-пути.
